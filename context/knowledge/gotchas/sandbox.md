## Sandbox (macOS sandbox-exec)

- **sandbox-exec uses SBPL profiles, not JSON.** `GenerateSandboxConfig()` returns `(profilePath, params, cleanup, err)`.
- **SBPL symlink trap:** All SBPL rules (deny AND allow) must use symlink-resolved paths — kernel resolves symlinks before matching. `evalSymlinksOrKeep()` resolves HOME, WORKTREE, git dir, and custom paths. Without this, rules containing `/var/folders/...` silently fail against kernel-resolved `/private/var/folders/...`.
- **Must allow writes to:** `~/.claude.json`, `~/.claude/` (or Claude hangs silently), `/var/folders` (macOS temp dirs), and the main repo's `.git` dir (for worktree git operations — `resolveGitDir()` handles this automatically).
- **SSH keys must be readable for git over SSH.** The profile does NOT deny `~/.ssh` reads — blocking SSH private keys breaks `git push`/`git fetch`. Since `(allow network*)` is already granted, denying key reads provides minimal security benefit. Other credential dirs (`.aws`, `.gnupg`, `.kube`, `.config/gcloud`) remain denied.
- **No domain-level network filtering.** sandbox-exec works at socket/address level. Argus uses `(allow network*)`.
- **Per-project sandbox config:** 3 columns on `projects` table. `ResolveSandboxConfig()` merges global + per-project. `agent.IsTaskSandboxed(task, cfg)` is the single-call helper combining resolution + availability check.
- **`task.Sandboxed` is persisted at creation time, not derived from current config.** The detail panel reads `task.Sandboxed` directly. Never re-derive sandbox state from config for display — it must reflect the setting active when the task was launched. All task creation sites (new task, review, fork, headless) must call `agent.IsTaskSandboxed()` and set `task.Sandboxed` before `db.Add()`.
- **Per-project `DenyRead`/`ExtraWrite` are only editable outside the TUI (DB or API).** The project form only exposes the Enabled toggle. Paths are preserved through form round-trips but cannot be viewed or cleared via the form. Deleting and re-creating a project erases its per-project paths.
- **SBPL has no `pseudo-terminal*` operation.** PTY device access requires explicit file-write rules: `(allow file-write* (regex #"^/dev/ptmx$"))` and `(allow file-write* (regex #"^/dev/ttys[0-9]+$"))`. Invalid operation names cause `sandbox-exec: unbound variable` at runtime. `TestGenerateSandboxConfig_ProfileValid` catches this by running `sandbox-exec /usr/bin/true` with the generated profile.
- **Must allow `lsopen` for OAuth browser login flow.** `(deny default)` blocks Launch Services. Without `(allow lsopen)`, `open <url>` fails with `LSOpenURLsWithCompletionHandler() failed with error -54`, preventing Claude Code from opening a browser for OAuth login/token refresh. Symptom: agent prompts for `/login` when token expires during a sandboxed session.
- **Nested sandbox-exec fails with exit 71 (`sandbox_apply: Operation not permitted`).** If the calling process is already sandboxed (e.g., running inside Claude Code's sandbox), `sandbox-exec` cannot apply a new profile. `sandboxExecFunctional()` detects this by running a trivial profile; `IsSandboxAvailable()` only checks the binary exists. All sandbox enforcement tests must use `sandboxExecFunctional(t)` and skip when nested.
- **`/var/folders` is allowed for writes in the sandbox profile** (`/private/var/folders` and `/var/folders`). This means `t.TempDir()` directories are writable under the sandbox — tests that need a write-blocked directory must use a path outside `/var/folders` (e.g., a subdir under `$HOME`).
- **Must allow writes to `~/Library/Keychains` for API key storage.** Claude Code uses macOS Keychain (`security add-generic-password`) to persist API keys. Without write access, `/doctor` reports "macOS Keychain is not writable" and login tokens cannot be saved — agents loop on `/login` prompts.
- **`~/.claude.json` rule must use `(prefix …)`, not `(literal …)`.** Claude Code (and most Node.js config persistors via `write-file-atomic`) saves config by writing `~/.claude.json.<random>` and renaming over `~/.claude.json`. A `literal` rule allows the exact final path but blocks every temp/backup/lock sibling, so the OAuth token write silently fails with EPERM and the agent loops on `/login` even though the explicit `~/.claude.json` rule looks correct. `TestSandbox_ClaudeJSONAtomicWriteSiblings` pins this — exercise `.claude.json`, `.claude.json.tmp.<rand>`, `.claude.json.backup`, `.claude.json.lock`, AND a negative case so a future over-broadening to `(subpath HOME)` is caught.
- **Codex backend needs `~/.codex/` writable — same EPERM-loops-on-login class as `~/.claude.json`.** Codex persists `auth.json`, session DB (`state_5.sqlite`, which `CaptureCodexSessionID` reads — so Codex must write it first), `history.jsonl`, `sessions/`, `models_cache.json`. Add the rule whenever Codex is a supported backend; the symptom is identical to the Claude bug. Pinned by `TestSandbox_CodexBackendWritable`.
- **`~/.ssh/known_hosts` must be writable (use `(prefix …)`) — otherwise ssh hangs the PTY silently on new remotes.** OpenSSH appends new host keys via `mkstemp` + rename (`known_hosts.XXXXXX`), so prefix not literal. Without this, `git fetch`/`git push` to a host not yet in `known_hosts` prompts interactively for `yes/no` and the agent's headless PTY hangs forever. Note SBPL `(prefix …)` is a pure string-prefix match (no path-separator requirement), so this also covers `known_hosts2` and any `known_hosts*` sibling — acceptable under the sole-user threat model. Private keys (`id_*`), `authorized_keys`, and `~/.ssh/config` remain write-protected — `TestSandbox_KnownHostsAppend` includes negative subtests pinning all three.
- **`~/.config/gh/` (gh CLI) is writable for token refresh.** Narrower than allowing all of `~/.config/` to preserve the `~/.config/gcloud` deny-read rule. `TestSandbox_GhConfigWritable` pins both the positive (hosts.yml write) and negative (gcloud read still blocked) cases.
- **Verify profile in effect:** Read the temp `.sb` file logged in `~/.argus/daemon.log`.
