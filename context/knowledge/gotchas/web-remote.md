# Web Remote Gotchas

Non-obvious invariants for the SPA + REST API + push notifications stack.

## Auth & EventSource

- **`EventSource` cannot set custom headers** — `/api/tasks/{id}/stream` MUST accept `?token=<token>` query-param auth as a fallback. The auth middleware (`internal/api/auth.go`) checks the Bearer header first, then falls back to the query param. Removing the query-param path will break the SPA terminal silently (the EventSource will 401 forever, looking like "stream just doesn't connect").
- **PWA assets must be exempt from auth** — `/`, `/vendor/`, `/sw.js`, `/manifest.webmanifest`, `/icon-*.png`, `/apple-touch-icon.png` are all unauthenticated. The browser fetches them at install/registration time before any login UI exists. Adding auth to any of these breaks PWA install on iOS without an obvious error.
- **Skip-paths ending in `/` match by prefix; otherwise exact match.** This is what lets `"/vendor/"` cover every vendor file via one entry. Don't strip the trailing slash thinking it's cosmetic.

## Offline view

- **Show after 2 consecutive `refresh()` failures, not 1** — a single 5s-poll miss is too jumpy for transient hiccups (laptop sleep, brief Tailscale renegotiation). `OFFLINE_FAIL_THRESHOLD = 2` in `index.html`. Two paths skip the threshold and call `showOffline()` directly: the initial `tryConnect()` (no cached UI state to keep) and the browser `offline` event (definitive). Once shown, `showOffline()` resets `connectFailures` to 0; subsequent polls still hit the catch and increment, but the `offlineShown` guard makes the re-call idempotent. Lowering the threshold makes the SPA flicker; raising it leaves the user staring at stale data without explanation.
- **`tryConnect()` distinguishes 401 from network failure** — `api()` throws `Error('unauthorized')` only on 401 (after calling `logout()`); any other rejection is a network failure. `tryConnect` checks `e.message === 'unauthorized'` to decide between the auth-error UI ("Invalid token") and the offline screen ("Cannot reach Argus"). Treating both the same was the previous behavior and made offline servers look like a token problem.
- **`showOffline()` tears down the SSE stream and detail view** — leaving an EventSource auto-reconnecting in the background while the offline screen is shown wastes battery and hides reconnect signal. `showOffline` calls `closeStream()` + `destroyTerm()` + collapses `#detail-view` and clears `currentTask`, so the user always lands on the task list when reconnect succeeds. Don't add a code path that flips `#offline-screen.shown` without invoking `showOffline()`.
- **`refreshTimer` keeps polling while offline** — the `setInterval` is started in both the success and offline paths of `tryConnect`. Without it, the offline screen would never auto-clear when the network comes back; the user would have to tap Retry every time. Don't gate `startRefreshTimer()` on connection success.
- **`logout()` must clear `refreshTimer` AND reset `offlineShown`/`connectFailures`** — without it, the post-logout `setInterval` keeps calling `refresh()`, which 401s with the empty token, increments `connectFailures`, and after two failures fires `showOffline()` *on top of* the auth screen. The user would log out and a few seconds later get an offline view they can't escape without retry → auth → retry. Whenever you add a teardown path that clears `TOKEN`, also stop the poll loop.

## Service worker

- **Never cache `/api/*`** — the service worker must use `network-only` for the API. Caching auth-bearing responses would either leak data or break dynamic state (task list, output stream). The `fetch` handler explicitly returns early for `/api/` paths.
- **`page.route()` cannot intercept fetches handled by an active service worker** — Playwright's network mocking sits below the SW. If a spec uses `page.route('**/api/**', route.abort('failed'))` to simulate offline, the SW's `fetch` event handler still dispatches `fetch()` itself and bypasses the route. Use `context.setOffline(true)` (browser-level) or invoke `window.showOffline()` directly instead. The same applies to any future spec that mocks API responses.
- **Rebuild `argus-test-server` after editing `static/*` — `go build` may serve stale embeds inside a worktree.** `internal/api` uses `//go:embed static/...`, but in a git worktree `go build` sometimes reuses the cached `internal/api` object even when an embedded file changed (likely a symlink-tracking quirk). Symptom: tests fail with "TypeError: window.foo is not a function" because the served HTML is the pre-edit version. Force-rebuild with `go build -a -o /tmp/argus-test-server ./cmd/argus-test-server` if `static/index.html` looks unchanged in the response body but the file on disk is correct.
- **Bump `SW_VERSION` on shell change** — caches keyed by version string. Without a bump the browser keeps serving the stale shell forever; users will think changes never deployed.
- **`sw.js` must NOT be aggressively cached** — the route handler sets `Cache-Control: no-cache` for `sw.js` only. Everything else gets `max-age=86400`. If `sw.js` itself caches, you can never push an update.

## xterm.js / SSE

- **base64 SSE chunks → `Uint8Array`, not string** — `term.write` accepts both, but using a string forces UTF-8 round-tripping that mangles raw bytes from agent output (status bars, ANSI sequences with high-byte content). Always decode base64 → byte array via `b64ToBytes`.
- **SSE writes must go through `bufferOrWrite`, not `term.write` directly** — iOS Safari kills momentum-scroll the moment JS mutates the xterm viewport, so any `term.write` while the user is scrolled into history (`!termIsAtBottom()`) cancels the active fling and the scroll dies after one line. `bufferOrWrite` queues bytes in `pendingChunks` (capped at 256KB FIFO) and `flushPending` drains them once `term.onScroll` reports we're back at the bottom or the jump-bottom button is tapped. Don't add a second SSE write path that skips this gate.
- **`termIsAtBottom()` alone is NOT enough — also gate on `termTouching` and a recent-scroll window.** Two race windows still kill iOS momentum even when the at-bottom check is in place: (1) between `touchstart` and the first `term.onScroll` event, viewportY hasn't moved yet so JS still sees at-bottom, and a write that lands here cancels the fling; (2) during momentum after `touchend`, scroll events keep firing for hundreds of ms, and the instant `viewportY` snaps back to `baseY` mid-momentum, a write races the still-active fling. `bufferOrWrite` checks `termIsAtBottom() && !termTouching && !inMomentum` (where `inMomentum = (now - lastTermScroll) < SETTLE_MS`); a `settleTimer` schedules an auto-flush `SETTLE_MS` after the last scroll/touchend. The touch listeners are attached in `setupTerm()` and MUST be removed in `destroyTerm()` (named function refs in `termTouchEl`) or they accumulate across open/close cycles and stale handlers toggle the flag for finger movements on a disposed term.
- **`syncVisualViewport` must defer during touch+momentum — its DOM writes mutate ANCESTORS of `.xterm-viewport`.** xterm puts the rendered rows in `.xterm-screen` (a sibling of `.xterm-viewport`), so cursor blink / row repaints are SAFE for momentum. The only mutators inside `.xterm-viewport` are `term.write` (grows `.xterm-scroll-area.style.height`) AND any DOM write on an ancestor (`<html>`, `#detail-view`, `.detail-header`). On iOS, `visualViewport.resize` fires mid-flick when the URL bar collapses on a fixed-positioned scrollable container; the resulting `setProperty('--app-height')` on `<html>` and `style.transform` on `#detail-view` race with the active fling and kill it. Gate with `if (termTouching || inMomentum) { pendingViewportSync = true; return; }` and have the settle timer drain `pendingViewportSync` after the same SETTLE_MS window. Reset `pendingViewportSync` in `destroyTerm`. Verify with a MutationObserver test if you ever change the listener set — `visualViewport.scroll` was confirmed harmless (no value changes), but resize is not.
- **`fitAddon.fit()` must run after layout** — call inside `requestAnimationFrame` or via `setTimeout(0)` after `term.open(...)`. Calling synchronously yields zero/incorrect cell sizes because computed CSS isn't ready yet.
- **Resize POST is debounced (100ms) on `window.resize`** — but use `setTimeout(200)` for `orientationchange` because iOS Safari fires `orientationchange` before the viewport actually settles.
- **Empty `[]taskJSON` encodes as `null`, not `[]`** — Go's `encoding/json` will emit `null` for a nil slice. Use `make([]taskJSON, 0)` for any handler whose response slice goes through the wire (`handleListTasks` does this; if you add another, do the same). The SPA does `tasks.find(...)` and crashes on null otherwise.

## HTML escaping in `index.html`

- **`esc()` only escapes `<`, `>`, `&` — NOT `"` or `'`.** It builds via `textContent` → `innerHTML`, which is safe in element-content position but injectable in attribute position when the attribute value is double-quoted. Patterns like `data-foo="${esc(name)}"` or `<option value="${esc(p)}">` are vulnerable to attribute escape via a `"` in `name`. For untrusted strings going into attributes, use `escAttr()` (which extends `esc()` by also escaping `"` and `'`), or prefer index-based lookup against a render-time array (see `renderedProjects` in `renderTaskList`). Project names, task IDs, etc. all flow from user-controlled DB rows.

## Detail-view layout

- **`#detail-view.open` is `position: fixed; height: 100dvh` so only the xterm scrolls** — page-level scroll inside the detail view fights xterm's scrollback for touch and used to make scrolling unusable on iOS. The replacement uses a flex column where `.term-wrap` is `flex: 1`. Don't reintroduce page-flow children that grow the detail view past viewport height; nested scroll comes back the moment you do.
- **Use `100dvh`, not `100vh`, for the fixed detail view** — Safari's address bar shrinks/grows the visual viewport but `vh` reflects the layout viewport, so `100vh` puts the bottom-anchored vkey-row behind the address bar / soft keyboard. `dvh` resizes with the visible area.
- **`dvh` does NOT account for the soft keyboard — wire `visualViewport` for that** — `dvh` reflects browser chrome only. When the OS keyboard slides up, `visualViewport.height` shrinks but `dvh` doesn't, so the vkey row gets hidden behind the keyboard. `syncVisualViewport()` writes `visualViewport.height` to the `--app-height` CSS var (consumed by `#detail-view.open { height: var(--app-height, 100dvh) }`) and translates the fixed view by `visualViewport.offsetTop` for iOS. Re-fit the terminal after the resize so xterm reflows for the new row count.
- **All action buttons live in the overflow menu (`#overflow-menu`)** — Esc, Toggle mode (Shift+Tab), Resume, Rename, Fork, Archive, PR and Delete are all menu items toggled via `#btn-overflow`. The button is pinned absolute in the top-right of `.detail-header` (no separate action bar). The IDs (`#btn-esc`, `#btn-mode`, `#btn-resume`, `#btn-rename`, `#btn-archive`, etc.) are preserved on the menu items so existing Playwright selectors still work, but tests must click `#btn-overflow` first to open the menu. `#btn-overflow` itself is in static markup and its click handler is wired once at script-load — don't try to attach it inside `renderOverflowMenu()` (that runs on every state change and would re-bind / leak listeners).
- **Mobile has no hard-stop — Esc replaced Stop in the menu** — Stop (which called `/api/tasks/{id}/stop` to kill the PTY) was removed; the `#btn-esc` item sends `\x1b` into the agent's stdin instead, which is a soft interrupt the agent honors most of the time. `/api/tasks/{id}/stop` still exists and is used by the TUI, but the SPA can't reach it. If Esc isn't honored, the user must switch to desktop. Items with `keepOpen: true` (Esc, Toggle mode) call `e.stopPropagation()` instead of `closeOverflowMenu()` so successive taps work without re-opening the dropdown.
- **Jump-to-input button visibility is driven by `term.onScroll`, not `term.write`** — the button appears whenever `buffer.active.viewportY < buffer.active.baseY` and hides when xterm auto-scrolls back to the bottom (which fires `onScroll`). Don't try to track scroll state from the SSE message handler — xterm's auto-follow already gets it right and `onScroll` covers manual moves.
- **No virtual key row** — Esc/Tab/Ctrl/arrows/^C/Enter were removed; the on-device keyboard supplies what's needed and modifier-only taps weren't worth the screen space. Font A−/A+ are the only key-style controls left and live as a row at the top of `#overflow-menu` (`.overflow-font-row`). Their click handlers `e.stopPropagation()` so taps don't bubble to the document-level click-outside handler that closes the menu, letting the user step the size repeatedly without re-opening the dropdown.
- **Header compacts when the soft keyboard is up** — `syncVisualViewport()` toggles `.detail-header.compact` when `window.innerHeight - visualViewport.height > 100`. The 100px threshold filters out browser-chrome shrinkage (address bar) and triggers reliably for both iOS (~250–340px) and Android (~300px) keyboards. Compact mode hides the back link + subtitle and shrinks the title so the terminal keeps the vertical space the keyboard ate. `closeDetail()` strips the class explicitly so it doesn't leak across navigations.
- **Flex children in the detail-view chain need `min-width: 0`** — `.term-wrap` and `#term` are flex items; without `min-width: 0` they refuse to shrink below their min-content width, so an oversized xterm canvas would push them past the viewport. The pattern matches the existing `min-height: 0` on the same elements; both are required for proper shrinking on phones.

## Web Push / VAPID

- **VAPID keys are persisted in the `config` table** — keys `push.vapid_public` and `push.vapid_private`. Regenerating them invalidates every existing subscription (the push service rejects with 401). Only delete these if you also clear `push_subscriptions`.
- **Push throttle map is in-memory only** — `idle:{taskID}` keys live in `Manager.lastSent`. Daemon restart resets all throttles, so a task that was just notified can be re-notified within 5 min. Acceptable for now; persist if it becomes annoying.
- **iOS Safari requires the page to be installed as a PWA before it will request push permission** — `Notification.requestPermission()` from a regular Safari tab on iOS silently denies. Add to Home Screen first, then open the standalone app, then tap Enable.
- **Push subscriptions returning HTTP 410 are auto-deleted** — `sendOne` checks for 410/404 from the push service and calls `DeletePushSubscriptionByEndpoint`. This is the only way to clean up after a user uninstalls the PWA.

## Per-device tokens

- **Master token is the only credential that can mint or revoke device tokens** — auth middleware sets `X-Argus-Auth: master|device` header on the request; mint/revoke handlers check for `master`. If you want to allow device-token-initiated minting, gate it behind a separate explicit capability flag, not just `auth != nil`.
- **`api_tokens.hash` is SHA-256 of plaintext, NOT bcrypt/argon** — bearer tokens already have ~256 bits of entropy from `crypto/rand`, so a single SHA-256 pass is sufficient. Don't switch to bcrypt thinking it's "more secure"; you'll add latency for no benefit.

## Settings endpoints

- **`PUT /api/settings` is partial — absent keys leave the value unchanged.** The handler dispatches to `buildSettingsUpdates`, which only emits a config write for fields whose pointer is non-nil. Slice fields use a separate sentinel: `nil` slice = leave alone, empty slice (`[]`) = clear. Sending `{"sandbox":{}}` is a no-op, not a reset.
- **Setting `kb.auto_start_todos` without `kb.auto_create_tasks` mirrors the TUI invariant: enabling auto-start implies auto-create; disabling it disables both.** This avoids the daemon silently falling back to fsnotify watching after a restart. If you let the SPA toggle auto-create independently, send both fields explicitly.
- **`requireMaster` checks the `X-Argus-Auth` header set by the auth middleware.** Hitting `srv.routes()` directly in tests bypasses that — every mutating settings/projects/backends test must wrap with `authMiddleware(srv.token, db, srv.routes())` or it gets 403.
- **`projectJSON.Sandbox.Enabled` is `*bool`** so `null` (inherit), `true` (override on), and `false` (override off) all round-trip distinctly. Using a plain `bool` would conflate "inherit" with "off". The SPA project editor renders three radios (Inherit / On / Off) and only sends a `sandbox` block when the user picked anything other than Inherit OR added paths.
- **`/api/logs/{name}` whitelists `ux` and `daemon`.** Anything else returns 400. Don't broaden by accepting arbitrary paths from the URL — `db.DataDir()` is HOME-rooted and a `..` segment would escape it.
- **Settings GET is readable by device tokens; PUT requires master.** This matches the existing pattern (devices can stop/start tasks but can't reconfigure the host). Read-only access is fine — every value GET-able from `/api/settings` is also visible in the TUI to anyone who can run argus locally.

## Test harness

- **`cmd/argus-test-server/` runs an isolated API server with `HOME=$tempdir`** — it MUST set HOME before any path resolution because `WorktreeDir()` and `db.DataDir()` resolve through `$HOME`. Without the override, the harness writes to the real `~/.argus/`.
- **Rebuild `/tmp/argus-test-server` after editing `internal/api/static/`** — `routes.go` uses `//go:embed static/...`, so `index.html`, `sw.js`, the manifest, and `vendor/*` are baked into the binary at compile time. `playwright.config.ts` uses `reuseExistingServer: false` and re-spawns the binary per run, but it does NOT recompile it — the cached binary's old assets ship instead. Symptom: Playwright fails with "X is not a function" referring to a JS symbol you just added, or assertions silently green against stale markup. `go build -o /tmp/argus-test-server ./cmd/argus-test-server` before each Playwright run that touches static assets.
- **`/test/reset` is a separate HTTP listener on `port+10`** — it's not under auth, so it must NOT be on the public listener. Tests call it from beforeEach to clear state between specs.
- **Playwright tests are NOT parallel** — `fullyParallel: false, workers: 1` in `playwright.config.ts`. The test server is single-tenant; concurrent tests would race on the seed task. Don't change this without rewriting the harness for multi-tenant isolation.

## Task status & idle

- **`taskJSON.idle` is true when DB status is `in_progress` AND there is no live session OR the session is `IsIdle()`** — a task with status `in_progress` and no Runner entry at all (e.g., daemon restart left the row InProgress but never re-started the agent) reports `idle: true`. Don't treat `idle && !running` as inconsistent — that's the load-bearing case the SPA needs to render the moon badge.
- **The SPA collapses `(status, idle)` into an `effectiveStatus` string** — `effectiveStatus(t)` returns `'idle'` for `in_progress + idle`, else the raw status. Anything that branches on status (`badge-*` class, `statusLabel`, sort `order`) must use `effectiveStatus`, not `t.status` directly, or idle tasks render as green "Running" while sorting in the actively-running bucket.
- **`s.running` from `/api/status` includes idle sessions** — the header shows actively-working count via `s.running - s.idle`. Don't display `s.running` raw; it overcounts by including agents that are sitting at a prompt.
