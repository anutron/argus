## UI Threading

- **Never run git commands synchronously on the UI thread.** Even fast commands take 50-500ms. Use background goroutines + `QueueUpdateDraw`.
- **The "no sync external commands on the UI thread" rule generalizes beyond git** — `launchctl`, `gh`, `go install`, anything that shells out. The tempting "<100ms is fine" carve-out is a trap: setting `busy=true` + calling `rebuildRows()` before the blocking call has zero visible effect because the tview event loop can't repaint until your handler returns. Pattern: settings view exposes an `OnXxx` callback, app wires it to `go app.doXxx()`, the goroutine calls `tapp.QueueUpdateDraw(SetXxxResult)` to clear busy on completion. See `OnToggleAutoStart` / `app.toggleAutoStart` for a minimal example.
- **Never call `GetInnerRect()` from the tick goroutine.** tview is not thread-safe. Store pending values under mutex in `Draw()`, read from tick goroutine.
- **`refreshTasks()` must not do RPC while holding `a.mu`.** Fetch `runningIDs` OUTSIDE the lock.
- **`TaskPreviewPanel.Draw()` must never call `runner.Get()` or create a VT emulator.** Pre-render in `RefreshOutput()` on tick goroutine; `Draw()` only paints cached cells.
- **Never run synchronous git commands on the tick goroutine.** Blocking the tick goroutine prevents `QueueUpdateDraw` from firing, freezing the UI. Use `go` + cooldown (e.g., `lastTaskGitRefresh` with 3s interval). The agent view already follows this pattern — the task list must too.
- **`onTick` must modify tview widget state only inside `QueueUpdateDraw`.** `TaskListView`, preview panels, and agent pane have no internal mutex. Direct modification from the tick goroutine races with `Draw()`/`InputHandler()` on the tview goroutine. Symptom: `buildRows()` sets `tl.rows = nil` then rebuilds — a concurrent `Draw()` sees the nil slice and renders an empty task list (project folders disappear).
- **Never call `QueueUpdateDraw` (or `QueueUpdate`) before `tapp.Run()` starts.** It deadlocks. tview v0.42 made `QueueUpdate` synchronous: it enqueues `{f, done: ch}` on the buffered `updates` channel (cap 100), then **blocks on `<-ch` until the event loop runs `f` and signals `ch` (a send, `update.done <- struct{}{}` at `application.go:512` — not a `close`)**. Without `tapp.Run()` running, no one drains the queue, the closure never fires, and `<-ch` blocks forever. Symptom: TUI starts (alt-screen sequences emitted), but no frame is ever painted, no log line after the queued call appears, and the process hangs until killed. Fire startup-time UI side effects (like opening the daemon-stale modal) by **calling the open function directly** before `tapp.Run()` — direct mutation is safe because no Draw goroutine exists yet (not because tview synchronizes; `Pages.AddPage` / `SwitchToPage` have no internal lock — only `Application.SetFocus` does). If you genuinely need the event loop, spawn a goroutine: `go a.tapp.QueueUpdateDraw(...)`. Earlier docs in this file claimed the buffered channel made queuing safe — that was wrong (tview's API changed).

## lazyScreen & Widget Painting

- **`lazyScreen` is a passthrough wrapper with no Clear-skip optimization.** A previous revision had `skipClear` to avoid the ~10K cell writes that tview's screen-wide `Clear()` triggered per PTY keystroke. That caused ghost status bars and tearing whenever the layout shifted between a skipped-Clear frame and the next real-Clear frame (resize, page swap, agent-header toggle). It was removed because `tcell.Show()` already diffs cells against their last-emitted state — `Clear()` followed by widgets restoring the same content produces zero terminal I/O, so the "optimization" was only saving in-process cell-buffer writes. Do not re-add `skipClear` without a strategy for invalidating it on every layout change.
- **Every widget that uses `DrawBorderedPanel` inherits an interior blank-fill.** `DrawBorderedPanel` calls `FillArea` on the inner rect with `(' ', StyleDefault)` before drawing the border. This is defense-in-depth — `tview.Clear()` already blanks the whole screen each frame — but it keeps widgets correct under any future Clear-bypass optimization.
- **If you add a new widget that does NOT go through `DrawBorderedPanel`, fill its full bounding box yourself in `Draw`.** `StatusBar` and `Header` already do this (they fill their row with a background-styled space). Widgets that only paint "occupied" rows (just the rows with content) break the moment a future optimization bypasses the screen-wide `Clear()`.
- **Layout-changing transitions must call `a.forceRedraw(reason)`.** `tview.Clear()` + diff-based `Show()` is insufficient in tmux/iTerm2/PTY-multiplexing environments — stale cells from the previous layout survive the Clear+Show cycle and leave visible ghost text (e.g., previous AgentHeader at top-right bleeding through to the Tasks tab; modal frame fragments lingering after `Pages.RemovePage`; doubled status bars after the agent header toggles). `forceRedraw` sets `app.pendingSync` (an atomic.Bool) which `SetAfterDrawFunc` consumes the same draw cycle and runs `screen.Sync()` — re-emits clear-screen + every cell, wiping the ghosts.
- **The "branch-change callback" contract — read this before adding a widget with multiple Draw branches.** Any widget that conditionally renders different content in the same rect (mode swap, tab swap, conditional row, scrollback toggle, diff view, banner overlay, focus-driven panel swap) MUST surface a callback the App wires to `forceRedraw`. tcell's per-cell diff is correct only when the SET of cells written is the same as last frame — branch swaps shift which cells are written and leave ghosts at positions the new branch doesn't touch. The structural hooks today:

  - `pages.SetChangedFunc` in `buildUI` — fires on every `AddPage`/`RemovePage`/`SwitchToPage`/`Show`/`Hide`. Covers every modal open/close, tab switch, and agent-view enter/exit (the `ResizeItem(header, ...)` calls in agent enter/exit don't fire SetChangedFunc themselves, but the `SwitchToPage` that immediately follows does — so the layout shift is captured).
  - `tasklist.OnLayoutChange` — fires when row signature changes OR filter-mode toggles (filter input reserves the bottom row; rowsSignature unchanged). Hashes kind + project + task ID + name + status; gate on this signature so the once-per-tick `SetTasks` rebuild and rapid `autoExpand` rebuilds don't Sync-spam.
  - `filePanel.OnLayoutChange` — fires when its row signature changes. Mirrors tasklist.
  - `agentPane.OnBranchChange` — fires on every TerminalPane Draw branch swap: SetSession (live↔nil↔replay), SetPending (banner↔normal), EnterDiffMode/ExitDiffMode/ToggleDiffSplit (PTY↔unified diff↔split diff), scroll-mode 0↔nonzero (live↔replay emulator), async replay rebuild completion (fallback→fresh emulator).
  - `taskGitPanel.OnBranchChange` / `gitPanel.OnBranchChange` (same widget, two instances) — fires when GitPanel's rendered branch changes: `!loaded` "Loading…" swap, the empty-state "Clean — no changes" swap, and section-presence flips (statusLines / diffLines / branchLines empty↔non-empty). `taskGitPanel` mutates on every task cursor move; `gitPanel` mutates on every agent enter/exit.
  - `taskPreview.OnBranchChange` — fires on SetTaskID change (clears cells, swaps to centered "Loading…"), `RefreshOutput` cell-nil↔grid transitions and grid dimension flips, and `SetStatus` placeholder-text changes. Per-cursor-move on the task list page.
  - `taskDetail.OnBranchChange` — fires when SetTask's shape signature changes: task ID, name, status, running flag, sandboxed flag, presence of Project/Branch/Backend/Worktree/CreatedAt/Prompt, and Prompt text content (different wrap → different row count). Elapsed time string width is intentionally NOT hashed — it ticks every second within the same task and the diff handles its width growth correctly.
  - Plus `afterDraw` detects screen-size changes (tview's EventResize handler does Clear+draw without Sync) and forces a Sync on resize.

  **Pattern for new widgets with conditional Draw branches:** expose `OnBranchChange func()` (not directly mutated by callers), wrap state-changing setters in helpers like `setFocus(f)` / `setMode(m)` that compare before/after and call `notifyBranchChange()` on actual change. Use the TerminalPane wiring as the template. Always add a smoke test that drives the branch swap and asserts the `[tui] force redraw: <widget> branch changed` uxlog entry.

  **Naming convention — `OnLayoutChange` vs `OnBranchChange`:** the two callback names mean the same thing structurally (the App wires both to `forceRedraw`), but signal different triggers to the reader. Use `OnLayoutChange` when the trigger is **row composition** changing inside a list-style widget (tasklist rows, filePanel directory expansion). Use `OnBranchChange` when the trigger is a **mode swap / focus swap / content-shape swap** within a single rect (TerminalPane diff vs PTY). When in doubt, prefer `OnBranchChange` — it's the more general term that the contract documentation uses. A widget can expose both (tasklist has `OnLayoutChange` for rows AND `OnFilterToggle` for filter-mode swap, with distinct uxlog reasons so a debugger can tell them apart).

- **`forceRedraw` is now flag-based, not channel-based.** Sets `app.pendingSync.Store(true)`. `SetAfterDrawFunc` runs synchronously inside `a.draw()` after `root.Draw(screen)` paints the new state but before `screen.Show()`; it does `pendingSync.CompareAndSwap(true, false)` and calls `screen.Sync()`. This eliminates the timing window the prior `tapp.Sync()` had — the queued Sync could run between events against a stale cell buffer. Multiple `forceRedraw` calls in one InputHandler / QueueUpdateDraw collapse to ONE Sync per draw cycle (idempotent flag). Safe to call from any context that produces a draw afterward (InputHandler, QueueUpdateDraw, paint callbacks). NOT safe to call from a goroutine that doesn't trigger a draw — the flag will sit set until the next unrelated draw consumes it.

- **`forceSync` is the multiplexer escape hatch, gated by a cell-buffer hash to avoid flashing on identical-content frames.** Set at startup by `detectMultiplexer()` (`$TMUX`, `$STY`, or `TERM` prefix `tmux`/`screen`). Overridable via `ARGUS_FORCE_SYNC=1|0`. When true, `afterDraw` hashes the cell buffer via `hashCellBuffer` (FNV-1a over content + fg/bg/attrs + underline style + underline color at every (x,y)) and calls `screen.Sync()` only if the hash differs from `lastSyncedHash`, OR if pendingSync / resize fired. Identical-content frames (no-op `QueueUpdateDraw` pings, idle ticks where nothing visually changed) short-circuit with no Sync. Rationale: `tcell.Sync()` sets `t.clear = true` which emits a `CSI 2J` clear-screen escape; tmux propagates that to the underlying terminal as a visible flash. Emitting it every frame produced the "heavy redraws" symptom users see in tmux. Hashing is O(W*H) plus one CellBuffer-mutex acquisition per cell inside tcell — sub-millisecond at 200x60 (12k cells); negligible next to the Sync it lets us skip. Layout-shift triggers still Sync unconditionally — those are the cases where tmux drift leaves stale cells and Show()'s diff isn't trustworthy, and `lastSyncedHash` is refreshed alongside so the next idle frame can short-circuit against the just-Synced state. The branch-change callbacks remain wired for bare-terminal users where the diff still wins (in bare-terminal mode `hashCellBuffer` is never called). Per-frame Sync is NOT uxlog'd — would drown the log. The startup log line `[tui] forceSync enabled — multiplexer detected, syncing every frame` is the only indicator. If you suspect forceSync is hiding a real bug (e.g. a widget that legitimately stopped repainting), set `ARGUS_FORCE_SYNC=0` and watch for the tear; conversely, if you see tearing in a bare terminal, try `ARGUS_FORCE_SYNC=1` to confirm the diagnosis before adding another OnBranchChange callback. **Known hash gaps:** `tcell.Style` stores OSC-8 URL + ID in unexported fields with no public getter, so a frame that changes ONLY a hyperlink URL hashes identical and skips Sync (Show()'s diff still emits the URL change so the live terminal sees it; only a subsequent tmux drift could leave the old URL behind). If you add a widget that mutates URL state on hash-identical content, wire it through an OnBranchChange callback so pendingSync fires.
- **Window-switch drift recovery uses tcell focus events.** `lazyScreen.PollEvent` intercepts `*tcell.EventFocus` with `Focused=true` and invokes the `onFocusGained` callback the App wires to `forceRedraw("focus regained")`. This closes the one drift scenario the hash gate cannot catch on its own: if tmux repaints our pane from a stale backing store while we were unfocused, no cell content changes on our side, so the hash matches and the hash gate skips the Sync. With focus reporting (`screen.EnableFocus()` called after `SetScreen` in `Run`), the focus-regain edge sets pendingSync and the next draw (≤1s via the tick loop) Syncs and clears the drift. The focus event is still returned for tview to handle (tview ignores it today but the lazyScreen wrapper observes rather than swallows). Don't call `QueueUpdateDraw` from `PollEvent` — tview v0.42's QueueUpdate blocks on a done channel and would deadlock since the polling goroutine is the same one that drains the update queue. Setting the pendingSync flag is sufficient because the tick loop guarantees a draw within 1s.

- **If `forceSync` does NOT fully fix tearing, escalate in this order — don't add more OnBranchChange callbacks.** The branch-change contract has been wrong six times (commits `9bb8d4c`, `1ff2bcc`, `b9baafc`, `0bc1cb0`, `b2bb42f`, `9a14af4`). The next two structural fixes, in increasing scope:

  **Option B — enforce "every widget paints its full bounding rect" as a debug-build verifier.** Wrap each leaf-primitive `Draw(screen)` call in a coverage check: snapshot the set of `SetContent` cell positions during the call, compute the primitive's `GetRect()` cell set, and assert equality. Once every widget covers its rect, tcell's per-cell diff becomes correct by construction (the SET of cells written is invariant, so Show()'s diff is well-defined) and the branch-change-callback contract becomes redundant. Cost estimate: 1 week. Trip points: defining "leaf" in tview's container tree (Flex/Pages don't directly paint), maintaining the parallel rect registry, performance hit in debug builds. Trigger: tearing reports from bare terminals (forceSync off, no multiplexer) where widgets visibly drop content from their rect.

  **Option C — centralize invalidation in one render-loop wrapper instead of N per-widget callbacks.** Replace the eight current `OnBranchChange`/`OnLayoutChange` callbacks with a single mechanism: hash the cell buffer slice of each leaf primitive's rect before/after that primitive's `Draw()` call; if the hash changed AND the rect's identity (XYWH or pointer) differs from last frame, mark dirty; at end of frame, Sync() if any rect was dirty. Cost estimate: 2-3 weeks. Trip points: stable rect identity across frames (especially under resize and Flex reflow), `fnv64` cost per primitive per frame (probably negligible), correctness of "dirty" semantics under stacked Pages. Trigger: tearing reports from bare terminals AND adding new conditional Draw branches has become routine (re-wiring OnBranchChange every PR). C makes the callback contract obsolete.

  **Why we landed on A (forceSync) first:** It's a ~10-line change that eliminates the entire bug class for tmux/screen — the environment where tearing has actually been reported. B and C are speculative for bare terminals where we have no failing reports today. Pick A → measure → only escalate to B or C if a real bare-terminal tearing report lands. Doing all three in one PR loses the bisect signal.
- **The previous tearing fixes (commits `9bb8d4c`, `1ff2bcc`) only patched two failure modes.** First, per-callsite forceRedraw missed half the modal close paths. Centralising on `pages.SetChangedFunc` made Pages mutations structural. Second, `rowsSignature` was extended to include name+status to catch task auto-rename and status flips. Both shipped, both still left tearing because the contract — "every branch swap must Sync" — wasn't codified beyond Pages and tasklist row composition. The four-hook structure above is the codification: filter toggle (caught by `setFiltering` notifying OnLayoutChange), file-panel expansion (caught by FilePanel.OnLayoutChange), and TerminalPane mode swaps (caught by OnBranchChange) all force a Sync. Adding a new widget mode? Wire OnBranchChange or OnLayoutChange and add a smoke test.

## Paste & Input Batching

- **`tapp.EnablePaste(true)` is required for fast paste.** Without it, tview delivers paste as thousands of individual `EventKey` events, each triggering a full screen redraw. With it, tview buffers all pasted text and delivers it as a single `PasteHandler()` call with one redraw.
- **`EnablePaste`/`EnableMouse` must be called AFTER `SetScreen`.** tview's `EnablePaste` only calls `screen.EnablePaste()` when `a.screen != nil`. And `Run()` only auto-enables when it creates its own screen (`a.screen == nil`). If `SetScreen` is called before `Run`, and `EnablePaste` was called before `SetScreen`, the flag is stored but `screen.EnablePaste()` is never invoked.
- **Every custom widget with text input must implement `PasteHandler()`.** tview's paste path bypasses `InputCapture` entirely — it goes through the focus chain calling `PasteHandler()` on the focused primitive. If a widget only has `InputHandler()`, paste is silently dropped when `EnablePaste` is on.
- **TerminalPane paste must wrap text in bracket paste sequences.** Send `\x1b[200~` + text + `\x1b[201~` so the agent's readline treats it as a paste (no per-character echo/processing).
