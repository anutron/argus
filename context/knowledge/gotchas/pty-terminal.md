## PTY & Terminal Rendering

- **`pty.Setsize` (which calls `os.File.Fd()`) races with `os.File.Close()`.** `Fd()` reads the internal fd field without synchronization, while `Close()` modifies it. `Read()`/`Write()` are safe (they use internal `fdMutex`). Fix: `ptmxClosed` bool flag under Session mutex; `waitLoop` sets flag + closes under lock; `Resize` checks flag under lock before calling `Setsize`.
- **`Session.Done()` blocks on `readDone`, not just process exit.** `waitLoop` waits for `readLoop` to drain the kernel PTY buffer before closing `done`. Without this, `Done()` fires while bytes are still in transit — `RecentOutput()` returns partial or empty data, and CI tests polling for output flake out (was the original `TestSession_RecentOutput` flake). The guarantee relies on `readLoop`'s `defer close(s.readDone)`.
- **`waitLoop` waits for `readDone` BEFORE closing `ptmx` in the fast path, with a `ptmxDrainTimeout` escape hatch.** Closing the master mid-Read interrupts the syscall with "file already closed" and discards any kernel-buffered bytes — exactly the bytes a short-lived child like `echo` produces just before exiting. After `Cmd.Wait()` returns, the caller-process copy of the slave fd is already closed (creack/pty closes it immediately after `cmd.Start`), so for a well-behaved child the master EOFs naturally within ms and `readLoop` exits on its own. If a grandchild inherited the slave fd (a Node subprocess without `stdio: 'ignore'`, say), the master never EOFs — `ptmxDrainTimeout` (5s) then triggers a force-close to keep `waitLoop` from hanging forever and leaking the task from `runner.sessions`. Reverse order = sporadic `TestSession_Attach_WithReplay` / `TestSession_RecentOutputTailWithTotal` failures on loaded CI runners.
- **PTY needs real terminal size at launch** (`pty.StartWithSize`), not 0x0. TUI apps won't render with zero dimensions. Start at actual panel width, not 80x24 — agents format initial output for launch PTY size.
- **Single-reader-tee pattern is critical.** Two goroutines reading the same fd causes data loss.
- **`AddWriter` must replay before registering.** Register first → live bytes arrive before replay → duplicate data → rendering corruption.
- **x/vt `SafeEmulator` hangs on terminal query sequences.** Use `newDrainedEmulator()` which starts `go io.Copy(io.Discard, emu)` to drain the response pipe. Never use bare `xvt.NewSafeEmulator()` in tui.
- **x/vt can panic on replay from differently-sized terminals.** Use `safeEmuWrite()` which wraps with `recover()`.
- **Cursor rendering respects `CursorVisibility` callback.** Tracked via `cursorVisible` field, updated by x/vt callback. Defaults to `false` on emulator creation.
- **Ring buffer must be bounded (256KB).** Unbounded causes CPU spikes and OOM. Session log file provides full scrollback.
- **Live scrollback reads from session log file, not ring buffer.** `readLogTail` seeks from EOF — gives infinite scrollback while live follow-tail stays on the fast ring buffer path.
- **Anchor-lock keeps scrolled-up content pinned when new output arrives.** Track `anchorTotalLines`; bump `scrollOffset` by the delta. Reset to 0 on scroll-to-bottom.
- **`asyncReplayRebuild` must signal `anchorTotalLines` reset via `replayRebuildPending`.** The replay emulator (fed from log tail) has far more scrollback than the live emulator (256KB ring buffer). Without reset, anchor-lock sees the totalLines jump as "new output" and bumps scrollOffset by the delta, causing a half-page jump on the first Shift+Up.
- **`ScrollUp`/`AccelScrollUp` must invalidate `replayEmu` AND `anchorTotalLines` on the 0→>0 transition.** Two bugs: (1) stale replay emu content — built during a previous scroll, its bottom doesn't match current live output, so `scrollOffset=1` shows content from hundreds of lines ago; (2) stale anchor — `anchorTotalLines` from live mode causes `paintEmu` to bump scrollOffset by `(replayTotal - liveTotal)`. Fix: `tp.replayEmu = nil; tp.anchorTotalLines = 0` when `scrollOffset == 0`, forcing a fresh rebuild from current log tail.
- **Replay emulator is cached and reused when only scroll offset changes.** Rebuild only when input size or dimensions change. For **live sessions scrolled up**, treat cache as always valid (new bytes arrive below viewport) — checking log size growth invalidates on every Draw since the agent is writing. For dead sessions, stat the log file (~1us) instead of reading it to check validity.
- **`replayEmuMaxScroll` must reflect actual emulator scrollback capacity, not the build-time scroll offset.** Setting it to `tp.scrollOffset` caused a full rebuild (log read + emulator feed) on every scroll step beyond the initial offset — the primary cause of slow scrollback in long sessions. Compute from `emu.ScrollbackLen() + lastContentRow + 1 - viewportHeight` after building.
- **Draw's `replayRebuildPending` consume block must clamp `scrollOffset` to the fresh `replayEmuMaxScroll`.** `ScrollUp`/`AccelScrollUp` can't clamp (no emu yet → no maxScroll), so the user can scroll past the available scrollback. Without snapping after a rebuild, the cache-validity check (`scrollOffset <= replayEmuMaxScroll`) fails every frame, each Draw kicks another async rebuild, and rebuild completion fires `notifyBranchChange` → `screen.Sync()` — visible as continuous flicker at the top of the scrollback. The clamp lives in Draw (main goroutine, under `tp.mu`), not in `asyncReplayRebuild` which is on a background goroutine and can't safely write `scrollOffset`.
- **Replay emulator rebuild must happen off the main goroutine.** `readLogTail` (up to 8MB) + `safeEmuWrite` blocks the main goroutine for hundreds of milliseconds with large session logs, freezing the UI because `QueueUpdateDraw` is synchronous. `asyncReplayRebuild` runs in a background goroutine; Draw() paints stale cache while it runs. The `replayRebuildPending` flag signals the main goroutine to reset `anchorTotalLines` and `paintCacheValid` (which are owned by the main goroutine). `readLogTailForTask` (not `readLogTail`) must be used from the background goroutine to avoid racing with `SetTaskID`.
- **Replay emulators use `newDrainedReplayEmulator` with 50K-line scrollback (vs 10K default).** This allows deep scrolling without rebuilds. The 8MB minimum log read populates ~60-70% of the buffer on first scroll.
- **First scroll-up must fall back to the live emulator while the replay builds.** `invalidateReplayCache` nils `replayEmu`, and `asyncReplayRebuild` takes ~1s for large logs. Without a fallback, the user sees "Waiting for output..." for the entire rebuild. `tp.emu` (10K scrollback) provides instant content. Priority: stale replay emu > live emu > placeholder. `scrollOffset`/`anchorTotalLines` must be saved/restored because the fallback emu's smaller scrollback may clamp them.
- **Agent view replay must use current panel dimensions, not stale PTY size.** Override `ptyCols/ptyRows` with current dimensions for dead/nil sessions.
- **Preview panel must use PTY dimensions (both cols AND rows), not panel dimensions, for VT emulation.** Width mismatch causes double-wrapping. Height mismatch causes blank space at top — ANSI cursor positioning formatted for a tall PTY gets clamped in a shorter emulator, leaving content in the wrong rows. `RefreshOutput` takes separate emu and viewport dimensions; the viewport selects which rows to display.
- **New emulators must default `cursorVisible` to `false`.** Agents send `\e[?25l` early, but after ring buffer wrap or emu rebuild, that sequence is lost. Defaulting to `true` causes a phantom cursor at bottom-left. Also, `lastContentRow` must not extend to cursor position when cursor is hidden.
- **`renderLive` must skip `RecentOutput()` when `newBytes == 0`.** The 256KB ring buffer copy on every draw causes typing lag — keystroke redraws trigger `Draw()` before PTY echo arrives, so copying is wasted. `emuFedTotal` must only advance when bytes are actually fed to the emulator; advancing it on an empty `raw` silently skips those bytes permanently.
- **`paintEmu` must cache cells for replay on idle redraws.** tview's `screen.Clear()` defeats tcell's dirty tracking (fills all cells with spaces → every `SetContent` marks cells dirty → full terminal I/O). On no-change redraws, replay cached `[]cachedCell` directly — skips 10K+ mutex ops, allocations, and style conversions per frame. Invalidate cache on scroll, reset, or session change.
- **`startAgentRedrawLoop` must skip `QueueUpdateDraw` when idle.** Keystroke and resize events trigger their own tview redraws; the 200ms loop only needs to fire when new PTY output arrives (`TotalWritten` changed).
- **`spinnerLoop` must only fire redraws for active (non-idle) running tasks.** Idle tasks show a static moon icon, not the spinner. Firing `QueueUpdateDraw` at 100ms when all tasks are idle causes unnecessary full-screen repaints that interfere with tmux hyperlink hover and waste CPU.
- **`uvCellToTcellStyle` must map ALL ultraviolet attributes.** Missing `AttrFaint→Dim` caused Ink-based CLIs (Codex) to lose visual contrast. Keep in sync with `uv.Attr*` constants: Bold, Faint, Italic, Blink, Reverse, Strikethrough. Also map underline styles (curly/dotted/dashed/double), underline color, and hyperlinks (OSC 8).
- **PTY keystroke follow-up redraw must guard on `TotalWritten`.** The immediate tview draw (from returning `nil`) fires before the PTY echo arrives. A 16ms delayed `QueueUpdateDraw` catches the echo; the `sess.TotalWritten() != tw` guard skips the redraw when no new output arrived, avoiding wasted work when the keystroke produced no echo.
- **tcell emits ~10K cells per keystroke as an unavoidable cost of tview's screen-wide `Clear()`.** After `Clear()` sets `currStr` on every cell, even a widget that re-SetContent's the identical original value gets marked dirty because `Put()` sees `cl != c.currStr` and calls `setDirty(true)` (clearing `lastStr`). A prior `lazyScreen.skipClear` hack bypassed `Clear()` entirely for PTY-keystroke frames, but it leaked stale cells on any layout shift between a skipped frame and the next real-clear frame, and was removed. Do not re-add it without a mechanism to invalidate the skip on resize/page-swap/layout-change — correctness beats peak typing throughput.
- **`RingBuffer.total` is `atomic.Uint64` — the only lock-free field.** All other RingBuffer fields (`data`, `pos`, `full`) require the caller's mutex. `TotalWritten()` is safe to call without any lock. `Session.TotalWritten()` and `RemoteSession.TotalWritten()` are lock-free.
- **`readLoop` data alias (`tmp[:n]`) requires all writers to consume synchronously.** Avoids a per-read heap allocation. Safe because `buf.Write`, `logFile.Write`, and all `io.Writer.Write` implementations copy or consume before returning. A future async writer that stores a reference to `p` would silently corrupt data.
- **`refreshPreview` TotalWritten guard fields must be protected by `a.mu`.** `lastPreviewTaskID` and `lastPreviewTW` are accessed from both the tick goroutine and `onTaskCursorChange` goroutines concurrently.
- **Never use `len(raw)` as a cache key for ring buffer content.** Once the ring buffer fills (256KB), `len(Bytes())` is constant — same length but different content on every wrap. Use `TotalWritten()` for change detection instead.
- **Panel-size delta is not a sufficient trigger for PTY resize.** `Draw()` only queues a resize when `ptyCols != wantCols`, but a stuck PTY (dropped SIGWINCH, daemon RPC timeout, session started in a smaller window) can have `ptyCols` already match the panel without Claude ever having received the new size. Agent-view entry calls `TerminalPane.ForceResyncPTY()` to repost the resize unconditionally on the next Draw. Keep the flag armed across dead/nil-session Draws so the next live session consumes it.
- **In tests, `bytes.Buffer` is NOT thread-safe with `Session.AddWriter`.** `readLoop` writes to every registered writer asynchronously; concurrent reads from the test goroutine race the writes even after `<-sess.Done()`, because there's no happens-before relationship between the writer-goroutine writes and the test-goroutine reads. `go test -race` flags this. Use the file-local `syncBuffer` (mutex-wrapped `bytes.Buffer` with `Write`/`Len`/`String` methods) in `internal/agent/session_test.go` for any AddWriter consumer in tests.
- **Don't `time.Sleep` after `<-sess.Done()` to drain readLoop.** Even 100ms is unreliable on CI under `-race`. The PTY drain happens in a separate goroutine that may still be running. Poll instead with a 2-second deadline, breaking when the expected output appears. Pattern: `for time.Now().Before(deadline) { if strings.Contains(string(sess.RecentOutput()), expected) { break }; time.Sleep(20 * time.Millisecond) }`. Both `TestSession_RecentOutput*` and `TestSession_MultiWriter` were converted from sleep to poll after CI flakes.
- **`computePTYSize` must NOT trust `agentPane.GetInnerRect` before tview has laid the pane out.** tview's `NewBox` returns a default rect of 15x10 when no `SetRect` has run, and `pw > 0 && ph > 0` doesn't reject it. The agent-view entry path is `SetSession → SwitchToPage → ForceResyncPTY → forceRedraw → startSession`, where `startSession` calls `computePTYSize` synchronously before tview's queued Draw can lay out the pane. After a daemon restart, the auto-start path read 15x10, derived a 20x8 PTY, started Claude narrow, and no SIGWINCH-triggered redraw could re-flow the already-emitted narrow cursor positions in the ring buffer (`\e[5;3H` still places content at col 3 even when re-emulated at 190 cols). Fix: compute from `term.GetSize(stdout)` first using the agent page's 1:3:1 layout ratio; fall back to the pane rect only when it's larger than tview's default (>30 cols).
- **`ptySizeFromHostTerm` deducts 4 rows, not 5 — the tab header is always hidden in agent view.** The row deduction is `agentHeader(1) + statusbar(1) + pane border(2) = 4`. Earlier code deducted 5 (including 1 for the tab header) as defense against the fork path running before the header was hidden, but every callsite (new-task `enterPendingAgentView`, auto-start `onTaskSelect`) hides the header BEFORE `computePTYSize` runs. Deducting 5 produced an off-by-one row mismatch: agent forked at `th-5` but pane inner is `th-4` once visible, so `forceResync` fired a Resize RPC ~200ms after the user reached the agent view, the kernel sent SIGWINCH, and Claude repainted — a visible content jump the user perceived as tearing. Fork still works: PTY at `th-4` exceeds the still-header-visible pane inner (`th-5`) by 1 row briefly, but the pane isn't on screen yet, and `onTaskSelect` hides the header before the user sees the agent view.
- **`SetSession` must seed `ptyCols/ptyRows` from the visible INNER rect, not `GetInnerRect()`'s outer rect.** `TerminalPane` is a bare `tview.Box` with no native border, so `GetInnerRect()` returns the outer rect of the pane; `DrawBorderedPanel` paints a 1-cell custom border inside it, so the actual content area is 2 cells smaller in each dimension. Seeding with the outer rect made `sizeChanged` fire on every agent-view entry (e.g. `192x84 -> 190x82`), triggering a Resize RPC and SIGWINCH that caused Claude to repaint visibly. Fix: subtract 2 from each dimension in the seed (with `w > 30 && h > 10` guard, matching `ptySizeFromPaneRect`'s threshold so tview's `NewBox` 15x10 default is rejected); Draw still self-corrects if the pane was reshaped between SetSession and the first frame.
- **Width-mismatched scrollback can only be repaired by kill+resume, not by SIGWINCH.** ink (Claude Code) re-renders live UI elements on SIGWINCH but the conversation scrollback is committed bytes — the cursor codes are baked in (`\e[5;3H` still places content at col 3 even when re-emulated at 190 cols). `Session.initialCols/initialRows` (set once in `StartSession`, exposed via `SessionInfo.InitialCols/Rows` and `SessionHandle.InitialPTYSize`) are immutable so callers can detect drift after the fact. The fix is bidirectional and fires from two sites: (1) TUI agent-view entry calls `maybeKickRerender`, sets `pendingNarrowRestart[taskID]`, and `handleSessionExitUI` immediately resumes via `--session-id`; (2) the API's `handleResize` calls `Server.maybeKickRerender` → `Runner.KickRerender`, which queues a daemon-side restart consumed by the runner's exit goroutine. Both sites use the shared `agent.ShouldKickRerender` predicate (margin = 30 cols, |panelCols-initCols| in either direction). Gated on `IsIdle()` to avoid killing mid-tool-call; gated on `task.SessionID != ""` because Codex captures its session ID post-exit and can't resume mid-conversation.
- **Daemon-side kick-restart must skip the InProgress→Review status flip and report Alive=true during the restart gap.** When `Runner.KickRerender` queues a stop+restart, the runner's exit goroutine fires `onFinish` (which would normally `transitionTaskOnExit` to InReview) before the new session is in place. The daemon's onFinish guards on `Runner.HasPendingRestart(taskID)` to skip the transition; without that, the row briefly flips to InReview and the API's `idleWatcher` / web SPA's status polling races the restart and pops a Resume modal. Symmetrically, `RPCService.SessionStatus` must report `Alive=true` when `HasPendingRestart` is set even though `runner.Get` returns nil — otherwise the TUI's stream client treats the gap as a real exit and tears down the session. The pending entry is held until the new `runner.Start` returns, closing the window.
- **`renderLive` full-replay path must read from the on-disk session log, not just the 256KB ring buffer.** Dimension change → emulator rebuild → original feed of `sess.RecentOutput()` (≤256KB) → x/vt re-emulates those bytes at the new width, but older history that was in the _previous_ emulator's 10K-line scrollback is dropped because it never entered the new emu. The live agent then re-emits a status bar at the new bottom row while the previous status bar — still in the 256KB ring — gets re-played and lands adjacent to it. Visible artifact: two status bars stacked at the pane bottom, no scroll indicator (still in live mode). Fix: `readLiveRebuildHistory` reads the 8MB log tail, merges with `RecentOutputTailWithTotal` to cover bytes that arrived between log read and snapshot, and feeds the result through `AlignToEscBoundary`.
- **Byte slices from arbitrary file/ring positions can start mid-CSI; align to first ESC before feeding the emulator.** `readLogTailForTask` reads `[fileSize - N, fileSize)` and the ring's `Bytes()` returns whatever wrapped-around position the buffer holds. Either tail can start in the middle of a `\e[5;3H` parameter list. x/vt's parser sees the orphan `5;3H` as printable text and renders a smudge of digits/punctuation at the top of a fresh emulator. `terminal.AlignToEscBoundary` skips to the first ESC byte; if there isn't one, the slice is treated as plain text. Apply to **every** fresh-emulator feed from a tail-positioned source: `renderLive`'s full-replay, `asyncReplayRebuild`'s replay-emu, and `TaskPreviewPanel.RefreshOutput` (preview panel rebuilds a fresh emulator every refresh from either `sess.RecentOutput()` or `LoadSessionLog`'s 64KB tail). The preview path went unaligned for months and produced visibly torn previews under long sessions — doubled status bars and orphan digit smudges in the agent preview rectangle.
- **Replay rebuild's `firstByteOffset` must be monotonically non-increasing across rebuilds for an alive session.** The default 8MB tail read positions `firstByteOffset = fileSize - 8MB`. When the agent writes more bytes between rebuilds, fileSize grows, so a naive re-read moves firstByteOffset forward and drops the oldest bytes the previous build cached. User-visible: scrolling past `replayEmuMaxScroll` triggers a rebuild → the deepest-cached content jumps out from under the cursor. Fix: `asyncReplayRebuild` accepts `prevFirstByte`, grows `needed` so the new firstByteOffset ≤ prevFirstByte, capped at `replayRebuildMaxBytes` (64MB).
- **`asyncReplayRebuild` must not trigger `onDone` when it bails out empty.** Calling onDone schedules another Draw, which immediately re-enters the slow path and kicks a fresh no-op goroutine — a CPU-burning spin loop while the placeholder is on screen. Set `replayNoDataUntil` to gate re-kicks for a short cooldown and skip `onDone` so external events (tick, keystroke, agent output) drive the retry.
- **`EagerReplayBuild` must skip when ptyCols/ptyRows are still zero.** Substituting 80×24 defaults produces a cache that the first Draw at the real panel size immediately invalidates (`replayEmuCols != ptyCols`). Defer the eager build until the first Draw has resolved the panel dimensions.
- **`paintEmu` must explicitly blank rows below the last content row.** When content shrinks frame-to-frame (cleared rows, scrollback contraction, dimension reduction), the row loop breaks early at `lineIdx > endLine` and leaves the trailing rows untouched. tcell's per-cell diff retains stale cells from the prior paint, and on the next keystroke redraw the paint cache (`replayPaintCache`) serves them again — visible as ghost lines below new content. Track `contentRows` from the row loop, then run a fill-with-spaces pass over `[contentRows, h)` and append those blanks to `paintCacheCells`.
- **`paintCacheCells` cap hint must include the scroll indicator's cells.** The scroll-mode badge appends 14 extra `cachedCell` entries after the row loop. Without `scrollIndicatorCells` reserved in `cacheSize`, every scroll-mode frame realloc's the backing array past the preallocation.
- **`tp.emu` rebuild must invalidate `paintCacheValid` before falling through.** Dimension change in `renderLive` creates a fresh emulator but the paint cache from the prior frame may still be flagged valid. The replay path (when no new bytes arrive but viewport matches) would serve stale cells for one frame before the next paintEmu refreshes. Set `paintCacheValid = false` inline at the rebuild branch in addition to the existing scroll/reset hooks.
