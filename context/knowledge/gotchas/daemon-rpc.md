## Daemon & RPC

- **Reconciliation must distinguish RPC failure from empty running set.** `Client.Running()` returns `nil` on RPC error vs `[]string{}` when daemon confirms no sessions. Check `runningIDs != nil && !daemonRestarting` before reconciling — during restart the new daemon has no sessions, so reconciliation would incorrectly mark all InProgress tasks Complete.
- **All daemon RPC calls must have a timeout.** `net/rpc.Client.Call()` blocks indefinitely. Use the `c.call()` wrapper with `select + time.After`. The buffered channel (`make(chan error, 1)`) is critical to prevent goroutine leak.
- **Stream loss ≠ process exit.** Use `StreamLost` flag to distinguish. When `isSessionAlive()` returns `reachable=false`, treat as stream lost, not exit. **The `<-rs.done` path in `connectStream`'s retry loop must also use `removeSessionStreamLost`** — `Client.Close()` during daemon restart closes `done`, and using `removeSession` causes a zero-value ExitInfo (StreamLost=false) because the GetExitInfo RPC fails on the closed client, incorrectly marking the task Complete.
- **`client.Get()` must return `nil` for any non-alive session.** Check `!info.Alive` alone — PID is stale between `onFinish()` and `delete(r.sessions)`.
- **Daemon health check uses `Ping()`, not `refreshTasks()`.** Ping fails fast; refreshTasks blocks on RPC.
- **Never call `refreshTasks()` from the tview main goroutine.** It blocks on RPC. Use `refreshTasksAsync()` (spawns goroutine + QueueUpdateDraw) or `refreshTasksLocal()` (reuses cached IDs, no RPC). Use `refreshTasksLocal` when only DB state changed (delete/prune); use `refreshTasksAsync` when session state may have changed.
- **Most `SessionHandle` reads on `RemoteSession` block on a `SessionStatus` RPC.** `PID`, `IsIdle`, `PTYSize`, `InitialPTYSize`, `WorkDir` all call `refreshInfo()` which fires the RPC. Don't call them from the tview main goroutine; do the read in a background goroutine and dispatch the result back via `QueueUpdateDraw`. Lock-free / local-only methods that ARE safe on the main goroutine: `Alive` (channel select), `Done`, `Err`, `RecentOutput`/`Tail` (local ring buffer), `AddWriter`/`RemoveWriter`. The `SessionHandle` interface doc enumerates this. `maybeKickRerender` was the regression that prompted this rule — it called `IsIdle()` and `InitialPTYSize()` from `onTaskSelect` and stalled the UI for the full RPC round-trip.
- **Never call `refreshTasksAsync()` between `db.Add(task)` and `startSession()`.** The async RPC snapshot captures running IDs before the session exists, then reconciliation sees InProgress + not-in-running-set → marks Complete. Use `refreshTasksLocal()` instead (no RPC, no reconciliation race).
- **`refreshTasksAsync` (and any other async RPC->reconciliation path) must capture `startGen` before the RPC and check it in the callback.** Without this guard, a session exit calling `refreshTasksAsync` while another task is starting captures stale runningIDs that don't include the new session — reconciliation marks it Complete. Same guard pattern as `onTick`.
- **Task creation must call `SelectByID` before `onTaskSelect`.** The ordering is `refreshTasksLocal()` → `tasklist.SelectByID(id)` → `onTaskSelect(task)`. Without `SelectByID`, the cursor stays on the previous row — when the user exits agent view, the task list highlights the wrong task. `SelectByID` must come before `onTaskSelect` because agent view covers the task list; the cursor position matters on return.
- **`startSession` must bump `startGen` before `runner.Start()`.** The tick goroutine captures `runningIDs` via RPC independently — if it snapshots before the session exists, the stale IDs reach reconciliation (which runs on the tview goroutine AFTER startSession). `onTick` captures `startGen` before the RPC; the `QueueUpdateDraw` callback checks if it changed and passes `nil` runningIDs to skip reconciliation for that tick. Next tick gets fresh IDs.
- **Always use `RunningAndIdle()` instead of separate `Running()` + `Idle()`.** They share the same `ListSessions` RPC — calling both separately doubles RPC overhead and doubles head-of-line blocking risk on the single `net/rpc` connection.
- **Daemon cleanup must run on the `Serve` goroutine, not `Shutdown`.** `Shutdown()` runs on a non-main goroutine — process may exit before cleanup completes. `signal.Stop(sigCh)` must be called after the handler goroutine exits.
- **Daemon restart must preserve `SessionID` on tasks.** Clearing it loses conversation history. On re-launch, `--resume <id>` picks up where it left off.
- **Rebuilding the binary does NOT update the running daemon.** Kill and restart: `kill -TERM $(cat ~/.argus/daemon.pid)`. The TUI also detects this on startup (via `Daemon.BootInfo` RPC + binary mtime compare) and prompts the user with a Restart/Skip modal — but only when it connected to a pre-existing daemon, not one it just `AutoStart`-ed (the auto-started daemon was forked from the same binary, so a check would always falsely show "in sync").
- **`Daemon.BootInfo` records `os.Executable()` + `EvalSymlinks` at `New()`, NOT at `Serve()`.** The mtime captured must reflect the binary the daemon process was loaded from. Capturing at `Serve()` is fine in practice but capturing at `New()` is more obviously correct since it pairs with `bootedAt`. Both run before any RPC can be served, so neither racing with the staleness check is possible. The argusd symlink in `~/.argus/` is resolved via `EvalSymlinks` so the recorded path matches what the TUI's `os.Executable()` resolves to.
- **`onFinish` must fire before session removal.** Callback runs OUTSIDE `r.mu` (to avoid deadlock), then `delete(r.sessions)` in a second lock section.
- **`startOrAttach` must revert all state on Start failure.** Reset to Pending, clear SessionID, zero StartedAt. `SetStatus(StatusPending)` doesn't clear StartedAt — explicit zero required.
- **Daemon session exits must be wired to TUI via `client.OnSessionExit`.** Without this, tasks stay InProgress forever in daemon mode.
- **`restartDaemon` must re-wire `OnSessionExit` on the new client.**
- **Fresh daemon auto-start AND daemon restart must set `daemonFreshStart=true`.** Reconciles stale InProgress tasks to InReview (not Complete) on the first tick. Without this on the restart path, tasks that lost their sessions during the restart cascade get incorrectly marked Complete. The flag is cleared after the first reconciliation so subsequent ticks use normal Complete behavior.
- **Daemon restart cascade: health check must have a 30s cooldown between restart attempts.** Without the cooldown, each auto-resume failure triggers another restart, creating a feedback loop (13 restarts in 40 seconds observed). The cascade kills sessions that were just started, causing orphaned processes and stale `connectStream` goroutines. **Both restart paths must set `lastDaemonRestart`** — the health check path AND the manual Settings `OnRestartDaemon` callback. Without setting it on manual restart, the health check can trigger another restart immediately (no cooldown).
- **`onTick` must read `a.runner` under `a.mu`.** The tick goroutine reads `a.runner` for `RunningAndIdle()`. During a restart cascade, `restartDaemon`'s `QueueUpdateDraw` writes a new client to `a.runner` on the tview goroutine. Without synchronization, the tick can use a stale client whose `ListSessions` returns empty → reconciliation marks all InProgress tasks Complete.
- **`Client.Close()` must signal `closed` channel before closing sessions.** `connectStream` goroutines check `c.closed` at the top of each retry iteration. Without this, old goroutines from a replaced client keep connecting to the new daemon's socket, flooding it with "session not found" requests.
- **Reconciliation must skip tasks started within the last 5 seconds (`recentStarts` grace period).** Belt-and-suspenders alongside `startGen`. After a restart cascade, `ListSessions` can return stale data if the RPC goes to an intermediate daemon. The grace period protects recently-started tasks from false reconciliation.
- **SessionID must be populated before first `runner.Start` for Claude backends, and captured post-exit for Codex.** Claude uses `--session-id <uuid>` on first run and `--resume <uuid>` on subsequent runs. Codex captures its ID from `~/.codex/state_5.sqlite` after exit. Without a SessionID, `resume` is always false and every start is a fresh conversation.
- **Codex session ID capture (`CaptureCodexSessionID`) must run off the tview main goroutine.** It opens a SQLite connection which can block. Use a background goroutine + `QueueUpdateDraw` to persist.
- **Self-update (`Daemon.UpdateSelf`) must NOT shut down the daemon itself.** The RPC only runs `go install`. The TUI calls `restartDaemon()` after a successful update, and the web API spawns a successor daemon inside a goroutine after the HTTP response is flushed. Shutting down inside the RPC handler would close the connection before the client receives the install output.
- **The web `POST /api/update` must flush the success response before spawning the successor.** Once the new daemon's startup runs `killExistingDaemon`, the current process gets SIGTERM and any pending HTTP write disappears. The handler writes the JSON, calls `Flusher.Flush()`, then sleeps 500ms in a goroutine before `exec.Command(os.Executable(), "daemon", "start")`.
- **`UpdateSelf` RPC must use a long timeout — the default 2s `rpcTimeout` will always trip.** `go install ./...` regularly takes 30s+ on a cold module cache. The client wraps `Daemon.UpdateSelf` with `callWithTimeout(...updateSelfTimeout)` (10 minutes); without this override the TUI returns `ErrRPCTimeout` even though the daemon-side build keeps running and eventually completes, leaving the user with a fresh binary on disk and a stale daemon still running.
- **`os.Executable()` after `go install` resolves to the freshly-installed binary.** On macOS `go install` writes to a temp file and renames it over `$GOBIN/argus`, so the path stored in the running process still resolves to the new bytes. `spawnSuccessorDaemon` relies on this to exec the new daemon via `exec.Command(os.Executable(), "daemon", "start")` after a successful update.
- **`onTaskSelect(task, autoStart)` auto-starts sessions when `autoStart=true` for any non-complete, non-archived task without a running session.** No explicit `startSession` call is needed after `onTaskSelect` — adding one causes a double-start. Pass `autoStart=false` for navigation-only entry (e.g., `navigateAgentTask`). The "user left pending view" path in `handleNewTaskKey` is the one exception (calls `startSession` without `onTaskSelect`). Condition excludes completed and archived tasks.
- **`selfupdate.Run` uses `git fetch origin master` + `git reset --hard origin/master`, not `git pull --ff-only`.** The source clone exists only to produce binaries — we don't preserve local commits or care about the current branch's tracking state. Using `pull --ff-only` made update fail with "no tracking information for the current branch" whenever the clone happened to be on a feature branch with no upstream. Reset-hard is destructive but intentional: whatever's on origin/master is exactly what gets installed.
- **`clipboard.Store` lives on `Daemon` and is shared with both the API server (HTTP + SSE subscribe) and the MCP server (`argus_clipboard_set` tool) via interfaces.** In-process TUI (no daemon) has no clipboard — the feature is gated by a `clipboardAccessor` type assertion on `agent.SessionProvider` and stays dormant when the assertion fails.
- **Subscribers in `clipboard.Store.Subscribe` MUST `defer unsubscribe()` immediately after registration.** Without it, an HTTP handler that returns through any non-loop path leaks the subscriber callback, and subsequent Set/Clear calls keep firing into a no-longer-drained buffered channel until it overflows and starts dropping (silently broken UI updates).
- **Daemon test names that hit Unix sockets MUST stay short.** macOS `sun_path` caps at 104 bytes; with the `/var/folders/...TestName123456789/001/test.sock` template, names longer than ~30 chars trip the limit and the daemon's listener silently fails to bind. `waitForSocket` returns "socket did not appear" 5 seconds later. Use abbreviations like `TestDaemon_Clip` instead of `TestDaemon_ClipboardRoundTrip`.
- **LaunchAgent `Install()` must bootout an already-loaded job before bootstrapping the new plist.** `launchctl bootstrap` errors with `Bootstrap failed: 37: The specified service is already loaded` if the label is already in the gui/$UID domain. The probe is `launchctl print gui/$UID/<label>` (exit 0 = loaded); on success bootout first, then bootstrap. Skipping bootout silently leaves the OLD plist active — the user thinks the rebuild is wired in but launchd is still running the previous binary path.
- **The LaunchAgent plist's `ProgramArguments[0]` must point at a stable path that survives `go install`.** Use the `~/.argus/argusd` symlink (created by `EnsureDaemonSymlink`), not `os.Executable()` directly. `go install` rewrites `$GOBIN/argus` via tmp+rename, so the symlink target stays valid; pinning the plist to the resolved binary path also works but locks the plist to whichever GOBIN was active at install time. Reinstall after relocating the binary.
- **`StreamHeader.Since` must be set to `rs.buf.TotalWritten()` on every stream attach, not just first connect.** Without `Since`, every reconnect (network blip, daemon restart, kick-restart gap, retry budget) replays the full daemon ring on top of bytes the client already has in its local ring — x/vt re-emulates the duplicated cursor-position sequences and renders a stale status bar overlapping the live one. The HTTP `/stream?since=N` endpoint already does this via `AddWriterFrom`; the daemon Unix-socket stream gets the equivalent by routing through `AddWriterFromTolerant` with `header.Since`. Client uses `rs.buf.TotalWritten()` (lock-free atomic) at attach time so retries skip bytes already received.
- **Use `AddWriterFromTolerant`, not `AddWriterFrom`, for net.Conn-backed writers (daemon stream).** `AddWriterFrom` holds `s.mu` through the replay so it can guarantee gap-free attach at exactly `offset` — but the constraint is that `w.Write` MUST NOT block, or readLoop stalls waiting on the same mutex. `conn.Write` to a Unix-socket peer can block on kernel socket flow control (typical 256KB send buffer, but partially filled if there are multiple slow writers attached). `AddWriterFromTolerant` mirrors `AddWriter`'s pattern: snapshot replay under lock, release, write replay, re-acquire briefly to append the writer — accepting a tiny gap (≤ one readLoop iteration's bytes) in exchange for non-blocking-writer safety. Use `AddWriterFrom` for the API's channelWriter (drop-on-full, non-blocking); use `AddWriterFromTolerant` for raw network conns. RemoteSession (TUI client) no-ops both since streaming is daemon-side.
- **`KeepAlive { SuccessfulExit = false }` is the right policy — `KeepAlive: true` would fight `argus daemon stop`.** With unconditional KeepAlive, every clean shutdown immediately respawns the daemon, making `daemon stop` and `daemon restart` racy. The dict form makes launchd respect zero-exit shutdowns while still recovering from crashes.
- **The LaunchAgent plist MUST include an `EnvironmentVariables` block that sets PATH explicitly, with `$HOME/.local/bin` baked in literally.** launchd starts processes with a bare `/usr/bin:/bin:/usr/sbin:/sbin` and does NOT expand `$HOME` inside the plist's `EnvironmentVariables` dict. When the daemon forks a child via `sh -c 'claude --resume <uuid>'`, the shell resolves PATH against whatever the daemon inherited — and the native `claude` binary installs to `~/.local/bin/claude` (Anthropic's installer default), which isn't on launchd's bare PATH. Symptom: every `--resume` (and every fresh `--session-id` task) fails with `exit status 127` and 31 bytes of output literally reading `sh: claude: command not found`, but only under the launchd-spawned daemon — daemons started from a terminal-launched `argus` work because they inherit the shell's PATH. The fix is in `renderPlist` (cover `<home>/.local/bin`, `/opt/homebrew/bin`, `/usr/local/bin`, plus the standard four); after editing, the user must reinstall the agent (bootout + bootstrap) so the new plist is actually loaded — the running daemon won't pick up the PATH change.
- **`inputLoop`'s drain MUST stop at bracketed-paste boundaries.** Two back-to-back `WriteInput` calls each carrying a complete `\x1b[200~..\x1b[201~` cycle (one paste per file when the outer terminal splits a multi-file drag-drop) get coalesced into one RPC and arrive at the PTY as one buffer; the inner CLI's bracket-paste parser then treats the second `[200~` as starting a fresh paste that replaces the first, dropping all but the last file. `drainInput` checks `bytes.HasSuffix(buf, "\x1b[201~")` and flushes immediately when the boundary is reached, sending each paste cycle as its own RPC. The fix preserves coalescing for plain typing.
- **Any helper that fork/execs `os.Executable()` with `daemon start` MUST refuse when running under a `*.test` binary.** Both `client.AutoStart` (TUI restart path) and `api.spawnSuccessorDaemon` (web `/api/update` path) check `isTestBinary()` and return a sentinel error (`ErrTestBinary` / `errSpawnFromTestBinary`). Without the backstop the forked child is the test binary itself — Go's test framework treats `daemon start` as positional args, runs `m.Run()`, and re-hits the test that triggered the fork, spawning another orphan. Two real fork bombs were observed (one in `internal/tui` from `TestApp_HandleRestartDaemonKey_RestartChosen`, one in `internal/api` from `TestSpawnSuccessorDaemon`), each pegging multiple CPU cores at ~85% and **overwriting the user's real `~/.argus/argusd` symlink** to point at a temp test binary that vanished when the test ended. Tests that exercise these paths must (a) override the call site (e.g. `App.restartDaemonFn`) where possible, (b) assert the backstop fires, and (c) still `t.Setenv("HOME", t.TempDir())` as defense in depth.
